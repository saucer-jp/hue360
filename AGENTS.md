# AGENTS

このファイルは、このリポジトリで作業する AI エージェント向けの恒久ルールです。
タスクごとの個別指示がある場合は、そちらを優先してください。

## Project Summary

- このプロジェクトは色相環ベースの配色確認・調整用 Web アプリです。
- 実装は Vite ベースのシンプルなフロントエンド構成です。
- エントリポイントは `public/index.html` と `public/js/main.js` です。
- ローカル起動は `npm run dev`、ビルド確認は `npm run build`、テストは `npm test` を使います。
- `main` ブランチが公開バージョンです。

## Source Layout

- `public/index.html`: 画面の土台
- `public/css/style.css`: スタイル
- `public/js/core/*`: 状態管理や色計算などの中核ロジック
- `public/js/views/*`: DOM 描画や UI 更新
- `public/js/utils/*`: 汎用ユーティリティ
- `vite.config.js`: Vite のルートとビルド出力設定
- `test/*.test.js`: Node 標準テストランナー用のテスト

## Working Rules

- 既存構成を尊重し、不要なフレームワーク導入や大規模リファクタは避けること。
- 小さく変更し、依頼範囲に直接関係しない箇所は触らないこと。
- まず既存コードの責務分割を確認し、`core` と `views` を混在させないこと。
- UI 文言やコメントを追加する場合は、日本語の既存文脈に合わせること。
- 命名は既存ファイルのスタイルに合わせ、不要な略語を増やさないこと。
- 既存テストがある領域を変更する場合は、必要に応じてテストも更新または追加すること。

## Branch And Release Flow

- 公開対象は `main` ブランチです。
- 日常開発は `develop` ブランチを起点に行います。
- 新規作業は `develop` から `topic` ブランチを切って進めます。
- ファイル編集、テスト追加、コミット準備を始める前に、現在ブランチを確認すること。
- 現在ブランチが `main` または `develop` のままなら、その場で実装を開始してはいけません。先に `topic` ブランチを作成してから進めること。
- `topic` ブランチを作成していない状態では、実装や repo tracked file の変更を行ってはいけません。
- `topic` ブランチ作成後、実装開始前に `develop` 向けのプルリクエストを作成します。
- 実装計画は、作業開始前にプルリクエスト本文へ記載します。
- 開発完了後は、作業内容と確認結果をプルリクエストのコメントへ記載します。
- レビュー完了後、プルリクエストを `develop` にマージします。
- リリース可能な単位の変更が `develop` に揃ったら、`develop` を `main` にマージします。
- `main` へ取り込むリリース時には、対応するリリースノートを作成します。
- リリースノート作成後、`main` の内容を Heroku にデプロイします。
- ブランチ運用やマージ先に迷いがある場合は、原則として `develop` を統合作業ブランチ、`main` を本番反映ブランチとして扱います。

## Development Procedure

1. `develop` を最新化して作業を開始します。
2. 作業開始前に現在ブランチを確認し、`develop` を起点にしていることを確認します。
3. `develop` から目的に応じた `topic` ブランチを作成します。
4. `topic` ブランチへ切り替わったことを確認するまで、ファイル編集を開始してはいけません。
5. 実装開始前に、`topic` ブランチから `develop` 向けのプルリクエストを作成します。
6. 実装計画をプルリクエスト本文に記載します。
7. 実装とテストを行い、必要な修正を完了させます。
8. 作業内容をコミットします。
9. 作業内容と確認結果をプルリクエストのコメントに記載します。
10. レビューを受け、必要な修正があれば追加で対応します。
11. レビュー完了後、プルリクエストを `develop` にマージします。
12. リリース対象が揃った段階で `develop` を `main` にマージします。
13. リリースノートを作成します。
14. `main` を Heroku にデプロイします。

## Work Start Checklist

- 作業開始時に現在ブランチ名を確認すること。
- 現在ブランチが `develop` なら、編集前に `topic` ブランチを新規作成すること。
- 現在ブランチが `main` なら、直接作業せず `develop` 起点の運用に戻すこと。
- 現在ブランチが既存の `topic` ブランチなら、そのブランチで作業すべき依頼かと、`develop` 向けプルリクエストが作成済みかを確認すること。
- ブランチ名は `type/short-description` 形式にし、作成後にそのブランチへ切り替わっていることを確認すること。
- プルリクエストを作成し、実装計画をプルリクエスト本文へ記載するまで、repo tracked file の編集を始めてはいけません。

## Branch Naming

- `topic` ブランチ名は、作業内容が分かる短い英小文字のケバブケースを使います。
- 形式は `type/short-description` を基本とします。
- `type` には `feature`、`fix`、`refactor`、`chore`、`docs`、`test` を使います。
- `short-description` には対象や目的を簡潔に入れます。
- ブランチ名に日本語、空白、大文字、曖昧な単語単体のみの名前は使いません。
- 例: `feature/add-harmony-preview`
- 例: `fix/slider-sync-bug`
- 例: `docs/update-release-flow`

## Implementation Guidelines

- ビジネスロジックや計算処理は `public/js/core` に寄せること。
- DOM 参照、イベント、表示更新は `public/js/views` や `public/js/utils/dom.js` 側に寄せること。
- CSS は既存の `public/css/style.css` を中心に拡張し、場当たり的なインラインスタイル追加は避けること。
- npm 依存は `import` で利用し、`public/js/lib` への手動同梱を新たに増やさないこと。
- 変更前に同種の実装を検索し、重複ロジックを増やさないこと。
- 将来の保守が難しくなる過度な抽象化は避けること。

## Validation

- コード変更後は、まず `npm test` を実行して既存テストを確認すること。
- 配信まわりの変更や依存更新を含む場合は `npm run build` も実行して、`dist` の生成成功を確認すること。
- 振る舞い変更やバグ修正では、可能なら再現ケースに対応するテストを追加すること。
- UI を変更した場合は、少なくとも関連する HTML/CSS/JS の整合を確認すること。
- リリース前は、`main` に入る内容が意図したものかと `dist` 配信前提の設定を確認したうえでリリースノートを更新すること。

## Safety

- ユーザーが作成した可能性のある未コミット変更を勝手に巻き戻さないこと。
- 破壊的コマンドの実行は避け、必要なら明示的な確認を取ること。
- 依存関係の追加、ビルド方式の変更、ディレクトリ構造の変更は、依頼がない限り行わないこと。
- `main` への直接作業や直接コミットは避け、原則として `develop` 起点のブランチ運用に従うこと。

## Response Style

- 変更内容の説明は簡潔に行うこと。
- 可能なら、何を変えたか、なぜ変えたか、どう確認したかを短く示すこと。
- 問題を見つけたが修正していない場合は、その理由や前提不足を明示すること。
